A function has multiple inputs (some optional) and multiple outputs.

Functions are connected to each other via lines. A single output can connect to zero more more inputs (if connecting to more than one, it means those paths can execute at the same time). You are not allowed to have multiple outputs connect to a single input (such behavior is difficult to define well). You're not required to connect any given output or optional inputs, (though at least one output must be connected or the function will never execute). Unconnected inputs will have, inside the function, fallback behavior that executed to supply that input.

Data-flow must be one-directional (i.e. a tree). You are not allowed to connect lines in a loop, i.e. no connecting an output line into your own input line. There are, however, tricks to recursively execute a function.

Either a single message must pass through a line, or the line will remain empty. When it's known that a line will remain empty, the "tension" effectively breaks on that line - it stops trying to look for an output there. If a function requires two inputs, and tension breaks on one of the inputs, it will still continue to pull on the other input, drawing out a value. Once it has the other input, it will break the tension to its outputs.

A future type system will likely force you to listen to enough outputs of a function as to guarantee that you're listening to one that will always execute, so execution doesn't abruptly and silently stop. If you want a line of execution to stop, you need to join it into an end node (the end node is a good pattern as that's a direct source for the tension required by execution).

Execution works bottom-up - by providing tension from the output side, causing the input "leaves" to execute to fulfill the outputs.

Syntax shorthands:
  If a function's output has been marked with "@primary", then you can pipe the output of that function into the input of another without creating intermediate variables. You can leave out the `->()` when doing this.
  If an input's name is "object", then you're allowed to use "." syntax to operate on it - the LHS of the "." will be passed in as the "object" input.
  If a function has a primary return, you can do `@return 'label' ...` to supply it. The default label is "out".

Useful derived functions:
  // The `fn` syntax is used to describe function behavior

  // A signal is a function that takes no inputs or outputs.
  Global Signal, Global toSignal <- ()
  // Define a signal. By setting the type (with the colon), we're forcing this function definition to conform to the type exactly - it can't
  // provide anything extra. It will also automatically have its inputs and outputs receive the input/output IDs as part of their names.
  def signal: Signal { ()->() }

  Global Tuple<T, U>, Global toTuple, Global fromTuple <- fn (left: T, right: U)

  // Combines left and right into a tuple.
  def tuple<T, U>(left: T, right: U)->(out: Tuple<T, U> @primary) {
    // Rename to avoid name conflicts (wouldn't technically be necessary since names use ids)
    left -> left_
    right -> right_
    def tupleInstance: Tuple<T, U>()->(left: T, right: U) {
      // Inputs are wired directly to outputs without calling functions first.
      left_ -> left
      right_ -> right
    }

    @return toTuple(tupleInstance)
  }

  // Splits the tuple up into two values.
  def untuple<T, U>(object: Tuple<T, U>)->(left: T, right: U) {
    fromTuple(object)()->(left=left, right=right)
  }

  // Messages are received from two inputs. A The second input is then forwarded to the output.
  // Both inputs must execute to completion before the output runs, and they both execute at the same time.
  def gate<T, U>(key: U, in: T)->(out: T @primary) {
    // I'm purposefully capturing `key` even though it's not needed - it'll force us to wait for the key to be brought in.
    @(key=key, in=in)->(fn=getResult)
    def ()->(out: T @primary) {
      @return in
    }

    @return getResult()
  }

  // Like gate(), but `in` won't evaluate until `key` has been received. If the `key` line breaks, `in` never evaluates.
  def lazyGate<T, U>(key: U, in: T)->(out: T @primary) {
    @(key=key)->(fn=getResult)
    def (in_: T)->(out: T @primary) {
      @return in_
    }

    // We can't evaluate the inputs to a function until we evaluate what closure we're calling (the closure
    // decides when to evaluate its inputs). And this closure won't get evaluated until it can capture a "key".
    @return getResult(in)
  }

  // NOTE: A function like this generally isn't needed.
  // The idea was to convert a value into a generic signal to trigger something else to happen,
  // but sometimes the function you want to call, moving around, can itself be a signal,
  // or sometimes you don't actually need a signal - you just need to call something with
  // an output and no inputs, and you cause things down the line to time when it gets called.
  def asNil<T>(in: T)->(out @primary)
    @return gate(key=in, in=getNil())

  def <T, U>join(v1: T, v2: U)->(out: Signal @primary)
    // Don't continue execution until both input paths have been received.
    @return gate(key=v1, in=gate(key=v2, in=toSignal()))
  
  // First the condition executes, then either `ifTrue` or `ifFalse`.
  def <T>if(condition: Boolean, ifTrue: T, ifFalse: T)->(out: T @primary) {
    fromBoolean(condition)->(out=bool)
    // The definitions for booleans haven't been provided, but it should be possible to define them so this works.
    // The `Boolean` function would be overloaded with a "true" and "false" input (you must provide one or the other)
    // This in turn causes either "true" or "false" to be captured, and then when we finally call it here, there's
    // two possible outputs but only one will generate a signal.
    bool->(ifTrue=ifTrueSig, ifFalse=ifFalseSig)
    // Because the two "outs" are from mutually exclusive paths, they can be joined in this fashion
    lazyGate(key=ifTrueSig, in=ifTrue)->(out=out)
    lazyGate(key=ifFalseSig, in=ifFalse)->(out=out)
  }
