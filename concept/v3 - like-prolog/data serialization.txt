You have nodes, defined via `def myNodeName`, and you have relationships, defined as `<node name>(key1=node1, key2=node2, ...)`.
Everything else is either syntax surger, or rules to enforce certain common behaviors within this framework.

```
// -- Core --
// `Node` is the base type. Everything is eventually a type of Node including Node itself. There's two main divisions underneath, Entities and Relationships.
// Rules would typically be added to enforce that only things of type Relationship can be used on the left-hand side of parentheses.
//   - Unless I can find a scenario where it makes sense to have something that behaves both as an entity and a relationship.
// Inside the parentheses you usually use other entities, but you can also use relationships.
// There will be other rules too. For example, for now, each node must have exactly one type relationship, and circular relationships aren't allowed (except for Node being a Node).
//
// Such rules could be defined elsewhere.
// If there's never a reason for someone to build a knowledge base that breaks these rules, then perhaps these sorts of things could be moved into a standard library
def Node // Base type
def type // `type` relationship

def Entity
type(target=Entity, type=Node)
def Relationship
type(target=Relationship, type=Node)
type(target=type, type=Relationship)
type(target=Node, type=Node)

// -- Entity Types --
def User
type(target=User, type=Entity)
def Report
type(target=Report, type=Entity)
def Permission
type(target=Permission, type=Entity)

// -- Relationship Types --
def username
type(target=username, type=Relationship)
def reportName
type(target=reportName, type=Relationship)
def reportAccess
type(target=reportAccess, type=Relationship)

// -- Entities --
// Each entity is really just a unique ids, I'm using names for clarity.
def userScotty
type(target=userScotty, type=User)
def myReport
type(target=myReport, type=Report)
def execPermission
type(target=execPermission, type=Permission)
def readPermission
type(target=readPermission, type=Permission)
def writePermission
type(target=writePermission, type=Permission)

// -- Relationships --
// keys are unique ids, but there should be friendly labels provided with them that an editor can show.
// (Original syntax was to do `theType { ... }`, but I switched to parentheses so `{...}` can be used for other purposes. Conceptually the idea is the same - these are like records)
// I'm ok letting two relationships have the same name if it's clear they have different signatures (e.g. a list having a "next" relationship is different from a number describing the next number, etc). Underneath there will be different ids for these two things with the same names.
username(target=userScotty, username="scotty")
reportName(target=myReport, username="my report")
reportAccess(user=userScotty, accessLevel=execPermission, report=myReport)
```

Undecided:
* Right now I'm treating types as just another relationship, and am causing a manual supertype relationship tree to be built. But if the relationship tree will always look like that, I could make it so types are more "special" where the data format itself requires a type to be provided everywhere, instead of separate rules within the data you have to add to cause it to be enforced.
* There's still open questions about how data inheritance might work, where "AdminUser" is like "User", but with additional properties - that sort of thing. I'm not sure if it's important, and I believe the flexibility of the way entities are associated with types should make anything that's needed to be possible, but it hasn't been explored yet.

Syntax shorthands.
  * BaseType, type, Entity, and Relationship are assumed to exist, you don't have to specifically declare them. (You can prefix them with `std.` if you want to make it clear that you're using a built-in definition)
  * You don't have to pre-declare entities and relationships. If you use it, we can auto-define it.
  * You can provide relationships to entities as you define the entities by using `with { ... }` afterwords. The `self` keyword refers to the entity being defined.
    def userScotty with {
      type(target=self, type=User)
      username(target=self, username="scotty")
      reportAccess(user=self, accessLevel=execPermission, report=myReport)
    }
  * For relationships that follow a simple `<relationship name>(target=self, <relationship name>=<value>)` pattern, you're allowed to shorten that to `<relationship name>=<value>`
    def userScotty with {
      type = User
      username = "scotty"
      reportAccess(user=self, accessLevel=execPermission, report=myReport)
    }
  * You can put the type before the opening curly bracket, as a shorthand.
    def userScotty with User {
      username = "scotty"
      reportAccess(user=self, accessLevel=execPermission, report=myReport)
    }
  * You can define sub-entities inline, like this:
      def thing1 with Thing {
        another = AnotherThing {
          ...
        }
      }
    This (currently) can only be done when the sub item doesn't need to have a name for others to reference it.

# Data Types

Natural number definitions:
  def zero with Natural {}
  def one with Natural { previous = zero }
  def two with Natural { previous = one }

  To negate a number to make integers, just do `def negativeTwo with Integer { absoluteValue = two, isNegated(isNegated=self) }`. Negative zero isn't allowed.

  You can use 1, 2, ... as a syntax shorthand for natural numbers.

  You can use +1, +2, -3, ... as syntax surger for integers.

Lists:
  def myListNode with ListNode { // Same as `[1, 2]`
    value = 1
    next = ListNode {
      value = 2
      // Note that `next=...` is omitted, that means this is the last item
    }
  }

  def myList with List {
    // If `head` is omitted, this will be an empty list.
    head = myListNode
  }

  You can use `[...]` syntax shorthand to define lists.

Strings:
  def myString with String {
    characterPoints = [64, 65, 66]
  }

  Syntax shorthand: `"..."`

Maps:
  def myMap with Map {
    pair(target=self, key="a", value="b")
    pair(target=self, key=0, value=1)
  }

  Syntax is `map["a" -> "b", 0 -> 1]`.

# Silos

Sometimes you may want to silo off an entire section of entities and relationships so they act independent from other ones.
Since everything has a unique id, you already don't have to worry about name conflicts if you stuff lots of data into the same file.
But you do need a way to know what data goes in what silo. Basically, you just need to associate all relationships and entities
to a special "silo marker".

```
// You would wire up external tooling to look for anything using this relationship
// (Note that there's nothing preventing two overlapping silos from being used on the same stuff, if wanted.
// I don't have a concrete use-case for that, but I also don't see a reason to try and prevent it)
def inDataSilo with Relationship {
  inDataSilo(self)
}

def execPermission with Permission { inDataSilo(self) }
def readPermission with Permission { inDataSilo(self) }
def writePermission with Permission { inDataSilo(self) }
def userScotty with User {
  inDataSilo(self)
  username="scotty"
  // Relationships such as `reportAccess()` would also be defined to be in the silo.
  reportAccess(user=self, accessLevel=execPermission, report=myReport)
}
def myReport with Report {
  inDataSilo(self)
  name="my report"
}
```

As a shorthand, you can just say that everything in the code block pertains to a given silo.

```
def * { inDataSilo(self) } // Indicates that all definitions that follow should have this relationship auto-applied.
def inDataSilo with Relationship {}
```


# Relationships as maps

I'm going to say that relationships are actually mapping from one entity to another. i.e.
  username(target=userScotty, username="scotty")
is really the same as
  { type -> username, target -> userScotty, username$2 -> "scotty" }
where all keys are entities and all values are entities. The `type` entity gets shared across all relationships. All other entities are, by default, not shared, and it should stay that way unless there's good reason to break the pattern (I don't enforce this at all, not sure if I should). (One of those is named "username$2" in the above example to disambiguate it from the "username" type).

This is useful for a few reasons:
* The "type" relationship isn't special anymore, it's just another key-value mapping
* Turns out there's other things besides "type" that benefit from being a shared key-entity used across multiple relationships. (I plan on adding ids to relationships when building rule syntax using this system)
* You can add metadata to keys by putting the keys into relationships

I think, going forwards, I'll start prefixing key names with "#" to help disambiguate them from other entities.
