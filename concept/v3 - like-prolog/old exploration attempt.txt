### Meta rules

Predicates are first-class. They can be treated as atoms and associated with each other. e.g. you might define and use a scope like this:
This scoping system really only works if your predicates are readonly
(are pairs with `invalid(true) :- ...` predicates that makes sure no one else is appending to them).
Special predicates used: $is(), $isPredicate(), $predicateContains().

isScope(isNumberScope)

isPublic(add)
readonly add :- ...

isPublic(equal)

readonly suc :- ...

// These are defined for the whole program

invalid(true) :- isPublic(X), \+ $isPredicate(X)
invalid(true) :- isPrivate(X), \+ $isPredicate(X)

// States that all predicates must be explicitly public or private.
// (Alternatively, you could default to private if it's not explicitly marked as public).
invalid(true) :- $isPredicate(X), \+ isPublic(X), \+ isPrivate(X)

// Ensures all predicates are part of a scope
// Note: I'm polymorphically calling these scope predicates.
//     If I ever make parameters take unique ids, I'll have to figure out how to handle this kind of scenario.
//     - This could be avoided if, instead of doing `myScope(myPredicate)` to associate a predicate with a scope,
//       I did `inScope(myScope, myPredicate)`.
invalid(true) :- $isPredicate(X), \+ (isScope(S), S(X))

// Something in the private scope cannot be accessed, unless by something else that's also in the same scope.
// $predicateContains() takes a predicate on the LHS and will tell you each predicate that's part of its body on the RHS.
readonly invalidViaInspection(true) :- (
  ($isPrivate(X), $isPredicate(Y), isScope(S), S(X), \+ S(Y), $predicateContains(Y, X));
  // Makes it so the only place $predicateContains() can be used is with this invalidViaInspection() rule. Other places can't be arbitrarily using it,
  // it makes it difficult to make private, internal APIs if this gets used willy-nilly.
  (isPredicate(X), \+ $is(X, invalidViaInspection), $predicateContains(X, $predicateContains))
)
invalid(true) :- invalidViaInspection(true)

Example of defining relationships based on rules:
  // Putting a rule on each line is another way of doing AND.
  bill -parent> alex AND alex -parent> bill
  // This relationship is not allowed to exist
  NOT sarah -parent> sam
  // Defining "or" - the items in angled brackets are the precedence
  def (`x` <5>OR<4.9> `y`) :- not ((not x) and (not y))
  // Both of these relationships are put in a sort of limbo state.
  // If you can prove that one of these is false, that will cause the other to be proved true.
  // You can either use the earlier rule, `NOT sarah -parent> sam`, or you could show that
  // no other rule exists that states that the relationship is true, so it must be false.
  // (The absence of a relationship is false)
  sarah -parent> sam OR sam -parent> sarah
  // §qcm4F
  // Conditional, or "implies". If X is true, then Y must be true. If X is false, ignore Y.
  //   Conversely, if I find out that Y is false, then I can deduce that X must be false as well.
  def (IF `x` THEN<10> `y`) precedence 10 right-associative :- (NOT X) OR Y

  // `ALL ... IN ...` is first-order logic's "FOR ALL", and is built-in
  ALL (x: Person) IN ALL (y: Person) IN ALL (z: Person) IN (
    IF x -parent> y AND y -parent> z
    THEN x -grandparent> z
  )

  // The `ANY `let x` IN<15> `y with x`` definition states that `x` is a variable definition, and its value will be made available to `y`.
  // This is the definition for "there exists" from first-order logic.
  // Note on how this works:
  //   EXISTS (P: Person) IN person1 -parent> P
  // is stating that some person is the parent of P. Who? Dunno. This is another way to put things into a limbo state until more
  // information can be found to resolve this further.
  def (EXISTS `let x` IN<15> `y with x`) :- NOT (ALL x IN NOT y)

  // Alternative formulation that may be easier to use
  def (EXISTS `let x` WHERE `y with x` IN<15> `z with x`) :- EXISTS x IN y AND z

    let n2: Number AND

What's described above is the lowest-level API surface area that can be shared between modules. Metadata can be optionally layered on top of it (such as "this is a primary return value"), then a "syntax engine" can be placed on top of that which lets you use easier-to-use syntax. The syntax engine may also provide support for macros that can also be imported and shared. These macros would be defined simply as data, but conforms to the syntax engine's plugin APIs.


How to define metadata?
  def sum(n: Number, m: Number, total: Number) :- ...

  The function itself would be represented as data. And it's easy enough to tack on metadata to the data as needed. With the text representation, we would probably add syntax for it, such as `def sum(n: Number, m: Number, total: Number) where isPrimaryReturn(%param:total) :- ...`
