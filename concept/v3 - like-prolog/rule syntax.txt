# Rule engine AST

To represent:
  parent(#parent=d, #child=e)
  for all (X: Person) in or(parent(#parent=a, #child=X), parent(#parent=X, #child=c))
Do:
  def * { inRuleMetaSilo(self) }
  def inRuleMetaSilo with Relationship {}

  def orBehavior with Relationship {}
  orBehavior(left=true, right=true, out=true)
  orBehavior(left=true, right=false, out=true)
  orBehavior(left=false, right=true, out=true)
  orBehavior(left=false, right=false, out=false)

  def or with BooleanOperator {
    // When the truth table gets used, the engine will be able to pick out certain behaviors that it should follow depending on the truth table.
    // For example, if this is an "implies"-type table and the LHS is false, it should realize that it doesn't matter if the RHS is true or false,
    // the result will always be the same. As such, from earlier notes, §qcm4F, it sounds like I would just ignore the RHS (as opposed to putting it into a limbo state).
    behavior = orBehavior
  }

  // Everything below this point will be marked as being in the "rule" silo.
  // Everything above is in RuleMetaSilo. RuleSilo uses things from RuleMeta silo.
  // (Perhaps "scope" would be a better name than "silo").
  def * { inRuleSilo(self) }
  def inRuleSilo with Relationship {}

  def parent with Relationship {}
  def a with Person {}
  def b with Person {}
  def d with Person {}
  def e with Person {}
  def $x with Variable { varType = Person }
  def rule1 with RelationshipId {}
  def rule2 with RelationshipId {}
  def rule3 with RelationshipId {}
  def rule4 with RelationshipId {}

  // All relationships will now have a new required parameter, relationshipId, that can be used
  // to chain them together.
  parent(#parent=d, #child=e, relationshipId=rule1)
  parent(#parent=a, $child=$x, relationshipId=rule2)
  parent(#parent=$x, $child=c, relationshipId=rule3)
  or(#left = rule2, #right = rule3, relationshipId=rule4)

  def relationships with RelationshipList {
    list = [rule1, rule4]
  }

# Minimal syntax engine

This is light syntax surger to make it easier to describe data encoded as described above. A heavier syntax engine will be needed as well, but that will be figured out later.

```
// 1. Boolean expressions can be nested
// 2. identifiers that start with $ will be considered variables, local to the rule.
// 3. `there exists ... in` and `for all ... in` are available. The `...` can be a comma-separated list.
there exists ($x: Person) in
and(
  left = parent(#parent=a, #child=$x)
  right = parent(#parent=$x, #child=b)
)
```

For now, you can't define custom boolean operators with the syntax surger, but there are these operators available:
  and, or, not, xor
  `if ... then ... end` is also available syntax for "implies". As a function, it's `then()`.
    (Some code examples leave out the `end` when it's obvious where it ends. Eventually I should stop doing that).
  `elseif ... then ... end` and `else ... end` are available too - they automatically NOT previous conditions in the chain.

You can also use the operators as operators:

```
there exists ($x: Person) in
parent(a, $x) and parent($x, b)
```

`there exists one ... in ...` is also available, and forces exactly one match. It is equivalent to the following
```
  there exists one ($a: Node) in <expr>
  // is the same as
  there exists ($a: Node) in <expr>
  not (there exists ($a: Node) ($b: Node) in (not $a is $b) and <expr>)
```

An `is` function/operator is defined as follows:

```
for all ($x: Node)
is($x, $x)

// Usage:
for all ($n: Natural)
if $n is 0 then ... else ... end
```
