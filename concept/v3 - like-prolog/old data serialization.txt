Mission?

"Foundation" is a proof engine that lets you codify high-level objectives, then separately provide an implementation description that's proven to obey the high-level objectives.
1. You provide information into a standard, but bare-bones format.

Philosophies:
1. High-level rules need to be "shelf stable", which means they should have few dependencies. (It's impossible to remove all dependencies, "Foundation" itself is a dependency, but it's intended to be a very long-lived dependency, hence the name).

-----------------------

Really, "Foundation" is about making something "perfect", meaning I feel confident in every design decision and have good rationals to back them all up, and none of the choices are related to current trends. As a test, I should try to at least make a "perfect" data-interchange format, if I can't even do that, then I'll need to lower my expectations.

Requirement:
  Additional metadata can be added at any time without making a literal or conceptual breaking change. ("conceptual" meaning the data would have been structured differently had the future been known from the start).
  Examples:
    A list of file names later needs to be treated as a list of files, so file sizes can also be returned. (XML forces you to wrap all list elements in a tag which prevents this flavor of the issue - is this a good solution?)
    "Inheritance", where you might have a list containing two entities of the same type, but you need to add more information to one of those entities that the other wouldn't support. For example, maybe the folder contains folders, files, and streams, where a file is a supertype of a stream (you can read/write to both files and streams, but you can also seek with files - maybe there's metadata about all of that you wish to communicate, depending on the type).
    "error graphs", where you can skip levels of inheritance when analyzing the entity.
    Your root data structure is a list, and later you realize you need other data as well
  Also consider the scenario where an endpoint receives a new parameter that, when set, causes something in the response body to always, or sometimes be a different type. This could be a conceptual breaking change. In the "sometimes" case, you now need to return a "type" field to distinguish the two possible data types while the original data structure didn't need to bother providing a way to distinguish itself. In the "always" case, you may still want to provide a "type" field, even though it's not strictly requires - it may be nicer for the end-user to handle either response payload type polymorphically instead of having to check if the flag was set to figure out how the response would look like.
  Can arbitrary rules be codified? E.g. a set of error codes could be returned, with the idea that more error codes may be added, and your program shouldn't break - i.e. you shouldn't be enumerating the contents of the set.

----- Proposal

The way you define data: You have a root record with various key-value pairs defined under reference number `$0`. Each key is identified by a unique ID and a friendly-name label. Code is required to look up entries by the ids, not labels (though ideally the editor would let you type out the label and invisibly auto-translate it to an id) *1 *2. Values are reference markers that point to definitions made lower down (or they can point back to the root, $0, to make a full cycle). You can provide multiple values to a given key.

Each id used should be "static" and have type definition information associated with it. I.e. you shouldn't represent a set by creating a record with various randomly-generated ids for keys. There's also no practical reason to do this - it's easy enough to just repeat the same id-key to represent a set.

```
$0 {
  68676 a: $1
  86597 b: $2
  86597 b: $3
}
$1 {
  01091 c: $2
}
$2 {}
$3 {}
```

I will be using the following syntax shorthands:
* Technically the labels are already a syntax shorthand. Those would be part of the type information, and your editor can automatically display them, but they're not part of the actual data.
* I won't bother providing explicit unique ids. You can assume that they're still being generated and used underneath.
* The `$0` can be omitted, it will be implied.
* You can nest records.
* You can prefix any record with a `$#` to define what its id should be, so others can reference the same record.
* You can use labels with `$`, such as `$userScotty`, to help make it clear what it is referencing.
* Syntax shorthand for numbers, strings, lists, maps, etc can be used, and will be compiled down as appropriate. (See `# Data Types` below)

So the above may also be specified as follows:

```
{
  a: {
    c: $1 {}
  }
  b: $1
  b: {}
}
```

Footnotes:
1. Alternatively, we could make it so only labels are used and you look up via labels, instead of having both labels with ids, and code mods may be provided if the labels change - this may be simpler to implement and use, but only works in scenarios where the receiver controls the version of the sender, such as a project to library relationship, it wouldn't work with a client to server relationship. If this alternative path is used, note that the same restriction as before still applies, but now needs to be more intentional - keys shouldn't be generated on-the-fly, so, for example, don't make a record act like a map (like you would in JSON).
2. This id + label design makes it tricky to use this data structure in any language except the one I'm designing. I'm going to be ok with that. "Easy for any language to consume" won't be a requirement.

Problem with data transfer is that the producer may be tempted to provide extra, duplicated data for convenience, such as calculating an age from a birthday and sending both. Really, what we should be considering is how data looks like when trying to construct a new object, then that objects public API can choose to provide duplicate fields. For data over network, assume that the client on the other end will use a library the data producer provides, that consume the data and produce an object that potentially has duplicate fields.

This design has private data from various object types being shown in a single tree. Generally it wouldn't work this way unless I needed to serialize it all.

------- Better description of the end-goal.

The ultimate point is to make a data file that multiple consumers can consume. I might not even need a "root object" in such a file as that would require all consumers to know how to parse the root.

Scenarios:
  An AST-to-JS converter needs to be able to see into the file, be given instructions on where to look, then find nodes that conform to its interface and create a compiled artifact.
  A proof engine needs to be able to do likewise.
  In general, the data saved will be a list of rules (however I decide to do that), but it should be possible to store other types of data into here as well, such as an alternative rule system than the default.

Note that three-way relationships are difficult with the current format. Giving a user exec/read/write access to a report would look something like this:

```
$userScotty {
  username: "scotty"
  reportAccess: { level: $read, report: $myReport }
  reportAccess: { level: $write, report: $myReport }
}
// Enums are being defined as numbers here. They could instead be random numbers, or better yet, symbols, but for now I don't support that.
$exec 0
$read 1
$write 2
$myReport { ... }
```

If we permit 3-way relationships, we could instead do this:

```
$userScotty {
  username($, "scotty")
  reportAccess($, $read, $myReport)
  reportAccess($, $write, $myReport)
}
```

Which feels more natural.

If I then make it so each parameter in a relationship is associated with a unique id (instead of a specific position), allow for optional parameters, etc, then these relationships start looking a lot like records. Which leads to "v2" (see next file).
