Categories of failures:
  * External issues: problems we acknowledge may happen while writing the code, and we're ok with the consequence of using the default failure handler.
  Examples:
    * Chosen assumptions violated, such as an assumption that data saved to a file won't be touched by anyone else, or the assumption of unlimited memory.
    * Code paths not written to handle certain scenarios, such as not bothering to handle network errors (it's not worth it for simple internal scripts where you can manually repair issues).
  * Bug: unanticipated issue that requires a fix in the code.
  Examples:
    * Logic error, caught by an assertion 
    * Incorrect type definitions for third party APIs

When a bug is detected, it means the system is in a bad state.
- If data was being touched, we should probably assume the data is in a bad state and human intervention would be required to fix it. In this case, good logging/notification is important, along with a capture of what the system was trying to do.
- If data wasn't being touched, then we should be able to cleanly kill a portion, or all of the process, without an issue. We still need to report the bug.


Scenarios: Installed GUI, installed CLI script, one-off CLI script (npx, or wget-and-run), webpage, server


-- On error reporting --

Bug reporting:
* Default to just showing the error information in stderr (or console.error() or a model for UI).
* Make error reporting customizable and completely replicable
* When reporting, do some de-duplication to make sure the same issue isn't reported more than once (though it would be good to count how often an issue happens).
* For client bugs, it's always best to give them the final say about if it should be reported or not - they may be running on a faulty disk drive and know it, or running a modified version of the code (or installed a mod), or may have monkey-patched the code. (Perhaps if I can figure out how to mitigate most of those issues, I could still auto-report as long as sensitive information isn't included, then provide a prompt afterwards, letting them explain steps to reproduce or to cancel the auto-reporting).
* Provide a low-fee, self-hostable service that provides a place for errors to report to.
* Provide good support for cluster stack traces - figuring out the path a message took across the cluster, across kafka inboxes, etc.
Don't bother planning the details out too much right now.



-- On error boundaries --

In general, if you're mutating memory, then either you need to show that the memory is in the same "error boundary" (so it would get wiped out if an error happened), or you need to interact with it as a side-effect.
  - side-effect based interactions would let you perform transactions against the memory, making sure multiple pieces of shared memory are updated at the same time and such.
  - Even if the memory is in the same error boundary, there's an advantage to still interacting with it via transactions. Probably the only exception would be some local variable where there's only one place writing to it.
  - Make sure to support things like memoization. That sort of behavior should work ok, since there's never a need to keep the data in a memoized function in sync with other data.

First and foremost, errors are bounded by events. Examples:
* The user clicks the "create" button and an assertion triggers - only the create action fails.
* A create-user endpoint triggers an assertion - that endpoint fails, but other endpoints may continue. (And if that endpoint was triggered by a button-push, the button-push action failed as well).
* The user runs a CLI command and something went wrong - the whole command will terminate (there's really nothing to salvage, the whole thing is one event - unless it was an on-going command such as running a web server).

Code can run in custom boundaries as well. This is needed when the child code path's success isn't required for the parent one to succeed. Examples:
* Writing to verbose logs. If that fails, that shouldn't cause the rest of the endpoint to fail.
* You trigger a shared-state cache refresh. This should be ran as a global user (not the user who performed the request) and counts as a "sub-event" you depend on. If the cache refresh fails, you could choose use the old cached value. (or choose to let the parent code path fail).
* If a component fails to fetch the required resources to run, it can be an error boundary, showing the error, but prevent it from propagating. This would happen during application initialization.

The above examples assume we had left all shared data (memory or disk data) in a good state and could continue where we left off. This might not be the case:
For example: A transaction may fail to complete if the latter part is simply buggy - e.g. if you're trying to write a message to an audit log endpoint at the end using invalid parameters, there's nothing you can do to salvage that situation. Things like saving your progress to the harddisk only helps if the endpoint was temporarily unavailable - if it is available but actively rejecting your request, that's an issue. We could still save off what the user was trying to do so when we report the issue, the developer can figure out how to repair the data - we might even be able to let the developer repair the function, then let these partial actions continue where they left off.
While we're in this partially completed state, we have to decide, in the code, what we're going to do about partial transactions:
* We could provide rollback behavior, but that's pretty ugly.
* We could lock down the transaction - we found it to be buggy, so we won't let anyone else touch it until it has been repaired.
* We could lock down any transactions that share data with the transaction we just locked down, if we suspect the data is in a bad state now. We could also lock down reads of that data.
  Locking down the transactions is quite blunt though, and could cause quite an outage depending on how widely used those transactions are..
* We could let the system continue to run as normal, and just generate a record of partially completed transactions that we will repair later
* Perhaps we provide ways to customize the error-handling strategies as well.
* None of these options are all that great - really, we just need to hope that user-intervention comes as soon as possible.

There are also events that may trigger that isn't based off of an end-user action, that is harder to just let fail.
* A periodic task (it's ok if these fail once, but if they repeatedly fail, that's an issue, and we may need to lock down the data they're supposed to help with).
* An "after user created" event - Causing this event to fail doesn't cause the entire user-creation event to fail, so we would be left in a bad state if this after-event fails.
The difference with these kinds of events, is that even if the very first item in the transaction fails, that could still mean we're in an invalid state, compared to most scenario (such as the user clicking the "create user" button - if the first write action there fails, the system is still in a good state).

-- "write" functions --

On transaction/write functions:
  In general, any time you're performing updates (of memory, disk data, network request that update, etc), that should be part of a transaction exposed to the caller. You should never have, e.g., a createUser() function that does various user-creation actions, that doesn't also let the user-creation process be part of a larger transaction. In other words, update-style functions are like async functions - they're part of the type signature that should effect all callers' type signatures.
  Due to the instability of end-users using a webpage, the way they perform transactions should be somewhat limited. For example, it's not possible for the client to safely start a transaction against two different servers or endpoints at the same time - the client could be interrupted after only one of those transactions got carried through. Instead, it's safer to send everything the client wishes to do to a single server, then that server can make sure all the steps get done, by distributing what needs to happen to its peers so someone can take over if it crashes, or by storing it in the harddisk so it can recover what it was doing after a crash. GUI applications are very similar, though they could also use the harddisk to keep track of the progress of their transactions. Though there's less guarantee that they'll start the application back up in a timely manner. Point is, environment matters, and the language may restrict what kinds of transactions you're allowed to perform depending on your chosen context, asking you to instead pass that transaction on to the server to perform.
  Exception: All write-functions are automatically joined together into a single transaction (perhaps with some explicit help to make sure it knows how to restore itself at that point in the code after a crash), but there should also be a way of separating it out, saying that these are two separate events that are allowed to fail separately - though that seems like it should be fairly rare.


~~ Transaction Definitions ~~

See also https://temporal.io/. It uses the terms "workflow" instead of transaction and "activity" for a single side-effect.

A transaction on a single-node machine requires recording progress on disk, so if it crash happens, it can pick up where it left off.
A transaction on a multi-node machine requires informing peers what it plans on doing (a PRE_TRANSACTION step) then executing each step, broadcasting progress as it goes along. If the node ever grows unresponsive, another node will take over and finish the remaining steps.
Also support: Custom ways to build up transactions (if something more efficient is needed), and not doing a transaction at all (e.g. an internal script, where repairs can happen manually if it goes wrong).
In general, you need to be able to describe:
  * Duplication handling: If we crashed in the middle of a step, should we re-do that step or assume it to be done (in other words, prefer it to be done at most once or at least once)? With an advertisement email, you might opt for at-most-once (it's not that important - better to not send it than to look buggy and send it twice). There's other options as well - you could additionally log out what got stuck in this decision, or make it so it doesn't auto-repair the problem and instead require human intervention, or if there's a way to check if the action got performed, you could try to do that, etc.
    This situation only matters if it's not idempotent. On a multi-node system, if we were passing an internal message around, we can de-duplicate repeated messages, so we don't have to worry about this issue. If the email-sending protocol could be updated to support de-duplication, that would be great, but the current reality is that it doesn't. Not everything can be de-duplicated - eventually you might be, for example, trying to write something to the harddisk (appending to a file) and you're trying to decide if that needs to happen again.
  * What data do you need to operate on? For example, you 1. Fetch the entire list of users, 2. Write out the length somewhere, 3. for each user do something. We crash between steps 2 and 3. Should the entire user list be written to a file (or broadcast) so we can continue step 3 just as planned? Or is it ok if a fresh user list is fetched? Or is custom handling needed? And what happens if you crash in the middle of step 3 instead - we would have to have recorded every user you were operating on so we don't re-operate on them.

I'm going to describe the way one interacts with transactions using an API instead of syntax. I can later find a syntax to wrap around them, but that's less important.

// This is all pseudo-code

// Transactions must be globally registered - you can't create any new transactions after the program's "main" has started running.
// Perhaps there's nothing against having a transaction factory as long as all of that executes before the "main".
Global .myTransaction = @transaction('myTransaction') (oldInboxId: string) => {
  // The `@save` annotation is used to mark data that should be saved to disk during transactions, so
  // if we need to recover the transaction, we can re-populate those variables.
  // You can do `@save<'xyz'>` to provide a custom ID for it for when it gets saved to disk.
  // The saved variable names must all have unique names.
  // The language will make sure a given transaction has everything saved that needs to be.
  //
  // This one uses `@maybeSave` instead of `@save` since it's the first variable save in the function,
  // and could perhaps be skipped if the only ones who use this function are other transactions.
  // Those other transactions would also be saving the data before it gets passed into here.
  // And since everything between these two points were pure, it wouldn't be hard just to re-run that bit.
  // Even if there was something impure between the two points, such as fetching data, it wouldn't be bad to re-fetch data.
  //
  // Alternatively, I could make everything before the first @step() act like maybeSave automatically - I'm not sure there
  // would ever be a reason to use `@save` at this point.
  //
  // Consider allowing a `@childTransaction` to exist as an alternative to `@transaction` which you can use if you
  // know the transaction will never be invoked in a stand-alone fashion. When `@childTransaction` is used, you
  // aren't required to save your initial data since it will already be saved for you.
  @maybeSave let inboxId = inNewFormat(oldInboxId);

  // Creates a recovery point. There can only be one `await` between each recovery point, unless explicitly marked otherwise.
  @step('listFiles', { repeat: 'atMostOnce' })
  @save let fileNames = await listFiles(INBOX_PATH + '/' + inboxId);

  @step('checkIntegrity')
  await dataIntegrityCheck(fileNames);
  if (fileNames.length == 0) {
    return;
  }

  @step('checkAllFiles')
  let results = await Promise.all(fileNames.map(fileName => {
    let result = await handleFile(fileName);
    @save fileHandlingResult = processResult(result);
    @step('checkAllFiles.singleFile') // ! Not exactly sure how this will work with describing a step inside a loop.
    return fileHandlingResult;
  }));

  // @runChildTransaction is like doing a @step() as well.
  // In reality, most of these `awaits` could probably be considered child transactions.
  // Not sure if you're really allowed to perform any side-effects without running transactions.
  await @runChildTransaction finishUp(results);

  return results;
}

// This can declare its own @save variables, even with conflicting names as before.
// It will work fine, because this is being ran as a child transaction that has its own name,
// so in the end, the save variables will really be saved as myTransaction.finishingUpTransaction.varName.
// I could have also defined this as a "partial transaction", which means it must be used as part of another transaction,
// and it will be treated as if it was inlined (partial transactions shouldn't be part of a public API as you can't easily make them stable - adding anything new would be a breaking change).
Global .finishUp = @transaction('finishingUpTransaction') (results: ...) => {
  ...
}

// When package updates happen, they can happen in a "transaction-breaking" flavor, meaning you're not allowed to make that update happen while you're in a transaction-recovery state. This isn't considered a breaking change, since it should generally be easy to get into a clean state before performing an update. But you can also push updates that are "transaction-preserving", which can be useful if you need to push out a repair for a buggy transaction.


-- Error handling and Backwards Compatibility --

Scenarios:
* Providing a JSON.stringify() function, then later adding a "strict mode" where you want really large ints to also throw errors on account of most clients not being able to consume them (unless you explicitly opt-out of this, because you know your clients can handle it).
* A file-opening function that now may throw a platform-unsupported error if the code is running in an only-in-memory OS (might not be a thing, but I'm pretending it is).

When depending on a library, there's (at least) two levels of dependency you can opt into. Level 1: Errors as "open unions". You will always provide a fallback error path to handle any other union variants (unless the library author specifically says a union is closed and any additions would be breaking). Level 2: Errors as "closed unions". You're allowed to depend on every union variant without providing a fallback path, but by so doing, you may receive breaking changes more often. Level 2 is necessary if you want to provide your own error messages or translations. Using level 2 should require explicit opt-in - we don't want end-users accidentally depending on fragile things for the first time.

Future: It could eventually be nice to give the authors AST transformation capabilities so they could make breaking changes, such as adding a new error union variant, but also provide a code transformer if they know how to make your code automatically handle the change so it behaves the same as before. (Consider having such transformers leave comments behind, indicating the code is shaped the way it is for backwards-compatibility, not due to an intentional decision). Example use-case - you used to escalate a particular type of error (such as out-of-memory or something), but decided to start handling it - ideally, existing consumers would be able to run a code mod and have their own code auto-escalate this new error type.
  Consider limiting the type of ast transforms that can be performed - doing so would make comments easier, as you wouldnt have to describe what exactly the comments are attached to - maybe.
  Also, not sure if I should allow fully automatic transforms - they perhaps should still be manual and reviewed?

-- Error tagging / Errors as objects --

Breaking an error up into further error types isn't a breaking change. e.g. before you returned a bad-parameters error - you still return that error type, but it now bundles multiple more-specific error types under it saying which parameter was bad. Be careful when doing this - this could technically be used as a way to introduce new error types, which could be breaking - if the user previously caught "bad parameter" and rewrote the error message to say one thing, but now "bad parameter" has an additional item under its umbrella, that end-user's rewritten error could become wrong.

Many of these comments talk about organizing errors as a tree, but I actually plan on using a tagging system.
  - Potential tagging use-case: You return a list of specific parameter-issue-errors under the "bad parameter" umbrella. If you want to make your library HTTP-friendly, you might also tag the issue as a parameter-not-found issue, so the consume could know to return a 404 error if it's a not-found error related to a specific parameter.

By default, all tree branches (including the root) will be considered open - consumer code should always assume that more could be added (and in general, that would be a best practice for consumer code - to not assume that just because there's only one reason for, say, a bad request error to be thrown, that it'll stay that way - unless documented otherwise and the type marks it as being closed - a leaf node that won't gain any children / a middle node that won't gain any more children).
  - A consumer can opt into treating them like closed unions at any point in time. They pick which one they want to fully exhaust, then do it, doing something to opt-into not having some kind of "if not any of these" branch.

Private error data: Errors could capture arguments and such that it holds private, not allowing anyone to view that information except your logging library. This would be unstructured data that could change at any moment, and shouldn't break anything.
Error data for interpolation: If a multi-step operation goes wrong, you may wish to provide the specific pieces of data that the consumer wouldn't have access to, but that they might wish to include in an error message.
Public error data: Use cases are really rare, but they're plausible.

The errors won't be actual unions (which means, perhaps I don't actually need to support "open" unions - not yet). Instead, they'll be objects. So, doAction() would return an actionFailed object. You can call methods to try and decern what failed, such as `actionFailed.badParameters()`, `actionFailed.userIdIsABadParameter()`, `actionFailed.notFound()`, and so forth. These return new error objects that may contain more data. Or something like that.
  I would need to heavily rely on macros to help generate these, and it would get messy combining errors from two sources. That being said, for internal error-passing, a simpler system could be used, like unions (which are always closed), and you switch to the fancier system at library boundaries - that may help encourage people to do their own error-translation if these fancier error types are easy to drill into but difficult to combine with other error types.
  These errors don't have identities - you're not allowed to check and see if the object returned from badParameters() is the same as the object returned from notFound(), because that detail may change. !!!!!!!!!!! How does that work - you wouldn't be able to provide type information for them, nor would the library be able to provide type information.

Here's what using error objects for the root node gives us:
  * At any point a concrete error can turn into a tag with child errors - we just add more functions to it to let you drill in further.
  * You can attach data, if needed, to any of these error objects.
  * These will be open-union by default - the API encourages you to assume that there could always be more drilling that could happen in the future.
  * Unions currently don't have a way to behave as an open union. I could add that capability, but right now it's not there.
