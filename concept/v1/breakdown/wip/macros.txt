An atom:
  Can be given to a function to grant permission to use a specific side effect.
  Can't be given to closures/classes.
  When a function is called with parameters, you know which parameters could hit the atom (i.e. you know what is tainted)
  Can be generic, for example, a "users table" atom-group may contain "read" and "write" atoms. You're only allowed to write data that fits the taints provided by the generic parameters, and when you read data, it'll be automatically tainted by the generics.
  New atoms can be made that restricts one or more existing atoms even further
  How do I support providing access to a glob pattern of paths? Or maybe even a predicate that returns true if you have access.
    - Whether I can call a function or not should be statically analyzable. class FileSystem {
  Global .new = () => new Class {}

  Global .FileSystemRead = class {
    FileSystem .new = (convert: <function>) => new class {
      Global .convert = convert;
    }
  }

  Global .FileSystemWrite = class {
    FileSystem .new = (convert: <function>) => new class {
      Global .convert = convert;
    }
  }

  Global (new Class).read = (path: String): void signals FileSystemRead => {
    // <-- This is all very messy. Is there a more stream-lined way to handle this?
    return yield FileSystemRead.new((): void signals IShell => {
      let subprocess = yield deps.shell.exec('cat ' + path);
      return subprocess.stdout;
    });
  }

  Global (new Class).write = (path: String, content: String): void signals FileSystemWrite => {
    yield FileSystemWrite.new((): void signals IShell => {
      yield deps.shell.exec('echo "' + content + '" > ' + path);
    });
  }
}

// <-- I don't currently support unions
Global .MainSigs = type IShell | IStdOut | IStdError

Global .main = (): void signals MainSigs => {
  const deps = {
    ...deps_, // <-- made up syntax
    fileSystem: FileSystem.new(),
  };

  try {
    yield helper(deps);
  } on signal (sig: FileSystemRead | FileSystemWrite) { // <-- I don't currently support unions
    // <-- 99% of the time, all the handler will do is "convert" from one signal type to another.
    // I don't love that arbitrary logic can happen in these "on signal" blocks as these aren't very visible - I prefer the logic happening
    // entirely inside these convert functions.
    let returnValue = yield sig.convert();
    sendBack returnValue; // `sendBack` controls what the completion-value of the "yield" was.
  } // Other `on signal ...` can be chained on as well, granting additional access.
}

Global .HelperSigs = FileSystemRead & FileSystemWrite;

Global .helper = (): void signals HelperSigs => {
  let text = yield fileSystem.read(deps, './in.txt');
  yield fileSystem.write(deps, './out.txt', text);
}

Advantages of signals being variables:
* You can break out of an outer-loop, JavaScript style.
* (minor) signals can be used in ternaries, as expressions, instead of being statement only.
* Due to the simplicity of skipping loops with breaking/continuing, even things like .map() could provide signals. For example, `array.map((x, mapSig) => x.subItem != null ? mapSig.continue : x.subItem)` lets me cleanly skip items as I'm mapping.
* May provide a solution to creating "holes" in arrays - e.g. `[1, condition ? holeSig : 2, 3]`. (though I'm not yet sure how to introduce the signal into the scope)
* It allows arbitrary callbacks to be interrupted (I don't have to provide a separate construct for a "block" that I pass into a loop that can be interrupted). (Though signals are probably not the only way to achieve this).

Is there ever a reason for something between the for loop and the signal-usage point to intercept the signal?
* There's reason to be able to accept a callback, then state that you are (or aren't) ok with that callback yielding signals. You might not be calling that callback right then, so you can't support signals passing through.
* You could run some kind of clean-up before propagating the signal. Though keep in mind that the function may be resumed (depending on if I support async/await or using signals for side-effects) - I may need a way to distinguish from temporary vs permanent interruptions if I support those things.
* Other than that, probably not.

What are signals:
* They're normal values with unique factories provided by a higher-order function, that, if produced, must be returned back to that same function. They cannot be consumed by anything else, so if you ever produce one from a factory, that signal must find it's way back to the higher-order function that provided the factory (it can't ever go ignored).
* They're branded, so you can check if something is a signal or not.

For an array.map() type definition, I need to be careful to make sure the obvious type for it of "mapping from one generic to another" doesn't automatically make it support signals incorrectly, where it'll put signals into the new array instead of returning them.
This is where "type families" come in. A generic parameter, by default, only includes types from the default family. Signals belong to a separate type family, and are only considered if explicitly called out.
NOTE: The "signal" type family has special semantics with it to enforce the "don't ignore instances" behavior. Only instances of Signal are part of this type family.

Global .forEach = <T, U is union of Signal of the SignalFam type family, V is an inferred signal instance of the SignalFam type family>(
  list: List<T>,
  cb: type (item?: T, doBreak: type () => V) => None | U | V
): None | U => {
  let head = list.headNode(); // Returns a maybe type
  if (head == Nothing) {
    return;
  }

  // If `maybeResult` is Nothing, then the signal was fired. If it's `Just<None | U>`, then it didn't.
  //
  // Signal.provideNewSignal() is in charge of the special constraints, making sure
  // any signals produced by the factory find its way back to the higher-order callback.
  // Future: Can this be built from other, more primitive tools? Can I provide syntax to help with this, especially to help with having multiple nested factories at once? This works, but it's a little clunky.
  let maybeResult = StdLib.Signal.provideNewSignal((doBreak) => {
    return cb(head.value, doBreak);
  });


  if (maybeResult.isNothing()) {
    return None;
  } else {
    let result = maybeResult.value();
    check result;
    return forEach(list.tail(), cb);
  }
}
