-- mixins --

A mixin is a value definition prefixed with `mixin`, such as `mixin value { ... }` or `mixin new Class { ... }`. You are allowed to mix them into other values with a spread syntax (`value { ...myMixin; }`). Some notes:
* Mixins are meant to be internal organization tools. The end-user cannot tell if you are using a mixin or not. (So unlike inheritance, a "mixed value" is not an instance of the mixin). If you need something like that, use interfaces (discussed below), which are capable of evolving separately from your mixin if needed.
  - Example: You could have a badRequestErrorMixin which contained fields common to a bad-request (such as the parameter name that had issues). You might also provide a BadRequest interface. The interface might choose to include the "message" property common to all errors while the mixin might not, since you might wish to mix multiple error-related mixins into the value, and there would be conflicts if they were all trying to provide a message property.
* If the target is a struct instance, then perhaps the source mixin should be required to be a struct mixin as well.
  Future: It does feel weird that you _must_ make a whole class-instance relationship to make a struct mixin. What if I just want a struct value that I can mix in?
* No field name conflicts (so no method overrides, super calls, etc)
* If a mixin doesn't define a field, then it can't refer to that field. (Again, different from inheritance where you could make an abstract base class that describes fields the subclass is required to define, that the super class will reference).

If you wish to mix both static methods and instance methods into a class, those would need to be provided by two separate mixins, individually mixed into the target. That could be organized by making a class that's a mixin, who's instances are also mixins - class mixes into class, instance into instance. Or it could just be two separate values.

It would be nice if editors showed you, on hover at a mixin sight, which properties are being mixed in.

-- interface --

These aren't traditional interfaces. They are made to fulfill a very specific role of sharing fields in a union, and allowing a union variant to be split up into further union variants without it being a breaking change. It works like this:

You have a function that returns type `A`, or maybe it returns `A | B`. You decide you want to be able to provide more information, so instead of an `A` class, you make an `A` interface, and then make classes `A1` and `A2` that implement the interface. Now the return type is either `(A1 | A2) &:A` or `(A1 | A2) &:A | B`. (The `&:` operator is a limited form of an intersection operator that is used to attach an interface to a union - I'm trying to avoid a real intersection operator as that has issues where the LHS type can't sprout new properties that conflict with the RHS). If you want to convert it to an open-ended union, you would instead do something like `(A1 | A2 | A) &:A`

Some notes:
* The "&:" is mostly an assertion so if you add new items on the LHS, the type system will make sure they conform to the type on the RHS - you won't accidentally change the interface in a way that breaks people's code. This is also one reason why it can be useful to not automatically share properties in a union without an opt-in like `&:` - making it explicit makes it easier to keep it stable (see §DCWN6).
* The instanceof operator cannot be used unless the union type explicitly says the instanceof operand is a valid target. So, for example, with `(A1 | A2 | A) &: A`, you can do an instanceof check for either `A1`, `A2`, or `A` - you can't go doing one for `A3`, even if it is an instance of `A`, and even though the type could be updated to include `A3` without it being a breaking change..

Question:

* Does the `(A1 | A2 | A) &: A` pattern actually work as a way to define an open-ended union? (Also consider adding "..." as syntax surger for auto-adding the thing after the `&:` into the union to make it open-ended). What if you wanted to make an open-ended union that didn't have a `&:` restriction?
* Is there a way for the language to enforce that if you have a function that returns `A3` instance and the return type doesn't include it, even if the return type is an open-ended union, ideally the language would help you remember to include it? Should it? Maybe it should be allowed to return some generic value that's not in the union.

-- What would it mean for all properties to be symbols symbols --

If you have a union, and all variants in that union have the same symbol key, then the language will allow you to access the symbol by name, e.g. `myObj.symbolName` becomes allowed. If you narrow down the union to exactly one value, then you can access any of its symbols by name. This is how `a.b` property access works.

You could have a union where some variants use one symbol with a particular name, and other variants use a different symbol with the same name - this is permitted, and you won't be allowed to access properties by name until you do further deductions.

An object is not allowed to have two symbol keys with the same name. Symbols are intended to help with unions, they are not intended as a means to introduce a trait/protocol system into the language.

Knowing that something is an instance of a class is sort of like an infinite union restricted by type. You would only be able to do property access using symbols you know are shared across all variants. Luckily, instances will automatically use the same symbol as each other, instead of creating a new set of symbols for each instance. Perhaps if you were to make a member that was private to the instance (instead of private to the class, the default), that would create a unique symbol per instance, which would prevent access to it unless you knew you were dealing with that exact instance.

Symbols are a new type of primitive produced via syntax: `#mySymbol`. (I would rather not use up the limited ASCII characters for this, but for now, I'll keep it). Each time the expression executes, a new symbol with that name is returned.

Example of using symbols - if Math and Calculations are used in a union, you'd be able to polymorphically use `sum`. I probably wouldn't allow `Math` and `Calculations` to be used in a union unless they also provide an identity (otherwise, what's the point), but we'll see, there could be valid reasons for error objects to be in unions without identities - if I want to be able to take one error type and split it into two, I can't let the originating error type have an identity or that would be a breaking change. -- misc --

This brings back a degree of polymorphism. But it's different - previous polymorphic designs were centered around having an arbitrary number of "types" that all conform to the same interface. This, instead, is centered around allowing a union to expose properties of its variants - these unions having a very limited number of "types" inside. This is also a user-case that polymorphism doesn't, by itself, solve, but it could - you could make a union definitions take an interface to say that "all variants in this union must conform to this interfaces, and all behaviors on these interfaces will be exposed" - e.g. `(A | B | C) of Error`. Something like that may be needed for open unions anyways.
