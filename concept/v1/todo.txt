Desired error-handling behavior:
  #drill-in: Errors can be organized into a tree (or graph?), letting you drill into the error type as far as wanted. (Is this a BadParameter error? Is this a UrlIsBad error?)
  #nonbreaking-growth: It should be possible to subdivide a tree/graph node further at any time without it being a breaking change. You can subdivide leaf or branch nodes.
  #open-closed: At every level of the tree, you can decide if it's closed or open. "closed" means introducing more categories at that level would be a breaking change.
  #drill-to-data: Errors can contain arbitrary data. As you drill in, more data may become available.
  #sharable: You can define multiple functions that return the same error "nodes", letting the end-user build error-handlers that can operate on those specific nodes.

You can't widen a type without opting into it.
  e.g. x: A | B = f() // returns A -- not allowed
  e.g. x:: A | B = f() // returns A -- allowed - double colon opts into the type widening -- or something like that.

  Due to the "no type changing" policy there really aren't many places that support type widening. Really it's just unions. And even then, perhaps it's better for unions to support the same policy.

  Scenario (document this): You have a third-party library function, `doThing()` that returns `A`. The end-user has a function with a body of `condition ? doThing() : Null`. Later, the third-party library makes a breaking change and `doThing()` now returns `A | Null`. You will never notice the breaking change - your function will automatically use the new `Null` type.
  Solutions:
    You have to explicitly opt into every union variant you wish to add? Seems verbose.
    Don't rely on unions so heavily. They can be a tool for third-party packages to provide errors from their public APIs, but then you just consume them, don't try morphing them around and such.

Open unions might need to be handles in a special fashion. Some principles:
  * `A | ... <- A | B | ...` should not be allowed. I don't want the open union to ever hide any actual stuff in the "..." bit - an open union is the ability to add stuff without it being a breaking change, it should not permit you to conceal stuff when you actually want to add it.
  * This means `A | ... <- A | ...` shouldn't be allowed either. If the LHS was the end-user and the RHS was the library, and the library later decided to add a new union variant, this code would break.
  * This effectively means open unions needs to follow the "no type changing" policy.

--------------- Unorganized thoughts

Libraries can be used, and they can be augmented. "Augmenting" is when you provide some kind of facade over it. Augmenting is when things get difficult, because there's going to be a lot of "inheritance"-like behaviors going on - you want a function that mostly accepts what the library can accept, but can also accept a few of your own parameters, or you want to return errors that may be translated in one way or another, etc.

If you don't know how things will evolve, it could be wise to not export the return type of your functions. If two functions return the same type, then later you want to add properties to one of those return types but not the other, that gets difficult. The language may still provide backward-compatible ways to handle it though, such as making one of the return types an intersection of two different types, `SmallerType & LargerType`, or by defining the larger type as being a superset of the smaller one (saying they can be used interchangeably).
